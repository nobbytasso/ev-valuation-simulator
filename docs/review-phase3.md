## 総評

6セクターのフォーム/結果ビュー/プリセット/ベンチマーク比較は統一パターンで実装されており、以下は良好:

- **metric_id網羅性**: ダミーJSONの全metric_idが「マッピング済み」または「理由コメント付きで明示除外」のいずれかに分類されている(除外4件: `pipeline_value_ratio`・`capex_intensity`・医療機器の`ev_sales_multiple`・`paid_conversion`)
- **エンジン純粋性**: `src/engine/` にDate/Math.random/DOM/ストレージ/Reactへの依存なし(grepで確認)。感度分析のドライバー変動・クランプ(U-15)もエンジン側に置かれておりUI都合のロジック混入なし
- **単位系**: フォームの%⇔小数変換、円(単価系)/百万円(金額系)の使い分けは6セクター一貫。EVレンジ・VC法の金額表示も統一フォーマット
- **テスト**: 6セクターとも同じ観点(プリセット適用/編集→保存/ベンチマーク差分/削除)+セクター固有の条件分岐をカバー

一方で、**計算の正しさ・データ互換性に関わる問題を3件**(D-1〜D-3)検出した。

## A. 要件からの乖離

### A-1. Comps企業の出典・取得日が非表示【中】

> §4.1.2「**全基準値に**出典・取得日を表示。ダミーデータ時はバッジ表示」

`BenchmarkComparisonSection.tsx` は `industry_standard` エントリのsourceのみ表示しており、`comp_company` エントリ(企業名・値はチャート上のマーカーとして表示)の出典・取得日が表示されない。「全基準値」の要件文言に対する乖離。

### A-2. ダミーバッジの「常時」性【中】

> CLAUDE.md 設計原則5「`data_status: "dummy"` のときUIに**常時**バッジ」/ §2 Stage 1「UIに『ダミーデータ』バッジ**常時表示**」

バッジが表示されるのはシナリオ詳細ページのベンチマーク比較セクション見出しのみ。シナリオ一覧・ポートフォリオページ・ヘッダーには出ない。「常時」をアプリレベル常設と読むなら乖離(ヘッダーに1箇所置けば充足)。

### A-3. IRR/MOIC出力が「含意IRR」どまり【中】

> §3 共通オーバーレイ「出力: 企業価値レンジ(悲観/ベース/楽観)、**IRR / MOIC**」

現状の `VcMethodSection` が出すのは「目標倍率(=目標MOIC)が含意するIRR」と必要持分の逆算のみ。**投資額×持分×Exit株式価値から計算した期待IRR/MOIC**は未表示。エンジン側には `irrBisection`/`moic`(希薄化シムと接続予定)が実装済みで、Phase 4の資本政策シミュレーター連動で充足される設計意図だが、要件文言上はシナリオ結果ビューの出力に含まれるため、Phase 3完了時点では部分実装であることを明記しておく。

### A-4. 並列比較ビュー未実装・フェーズ未割当【中】

> §4.1.1「シナリオ管理: 作成・保存・複製・命名・削除、**並列比較ビュー(表+チャート)**」

CRUDはPhase 2で、セクター別ビューはPhase 3で実装済みだが、複数シナリオの並列比較ビューはどのPhaseにも割り当てられておらず未実装。§9のフェーズ表にも明示がない(Phase 4の感度分析、Phase 6の磨き込みのどちらに寄せるか要決定 → D-11)。

### A-5. 金額表示単位の切替未実装【低】

> §1「金額は百万円単位を基本、**切替可**」

百万円固定。切替UI(億円等)はどのPhaseにも未割当。

### A-6. スキーマ提供情報の一部未表示【低】

スキーマにある `basis`(値の根拠)、`notes`、セクターレベル `as_of`(データ基準日)がUIに出ていない。§4.1.2の出典表示の趣旨からは補足情報だが、`basis` は「直近通期実績」等の比較妥当性に関わる情報のため表示価値が高い。

### A-7. metric_id命名契約の不一致【中】

> `benchmark.schema.json`: metric_id「**エンジン側ドライバーIDと一致** (例: arr_growth_yoy)」

実際にはエンジンのドライバーはcamelCase(`arrGrowth`)、ベンチマークはsnake_case(`arr_growth_yoy`)で、文字通りには一致していない。UI側のマッピング表(`{sector}BenchmarkMetrics.ts`)が橋渡ししており動作上の問題はないが、スキーマが謳う契約と実装が食い違っている。Coworkプロジェクト(実データ作成側)がこの文言を信じてエンジンコードを参照すると混乱する。→ C-8/D-10

## B. セクター間の実装不整合

### B-1. 感度分析ドライバー定義が創薬のみ欠落【中→Phase 4ブロッカー】

grep確認の結果、`SENSITIVity_DRIVERS`+`apply{Sector}Driver`+`{sector}BaseEv` の3点セットは saas / ecD2c / mediaTech / medicalDevice / climateTech の5セクターに存在するが、**drugDiscovery.ts のみ存在しない**。Phase 4(トルネードチャート)着手時に必須となる。創薬はネスト構造(assets配列内のドライバー)のためdriverId設計が他セクターと異なる工夫を要する点も先に決めておくべき。

### B-2. keyMetrics表示粒度の不統一【低】

| セクター | エンジンが算出 | UIが表示 |
| --- | --- | --- |
| SaaS | ruleOf40 | ruleOf40 ✓ |
| EC/D2C | ltv, ltvCacRatio, contributionMarginRatio | ltvCacRatio, CM率(ltv円値は非表示) |
| メディア | avgLifetimeMonths, ltv, ltvCpaRatio, paybackMonths | ltvCpaRatioのみ |
| 医療機器 | (なし) | ― |
| クライメート | (なし) | ― |
| 創薬 | (なし) | POS(UI側で独自計算) |

Phase 2の`GenericScenarioView`は全keyMetricsを機械的に表示していたが、専用ビュー化で各セクターが恣意的なサブセットを表示する形になった。特にメディアの`paybackMonths`(CAC回収の直接指標)が非表示なのは §3.4 のドライバー趣旨からもったいない。

### B-3. 派生指標のUI二重実装【中】

エンジンの計算式をUI側で複製している箇所が3つある:

1. 創薬 `DrugDiscoveryScenarioView.tsx` の `computePos()` — engine-spec §2.2のPOS式の複製
2. 医療機器 `medicalDeviceBenchmarkMetrics.ts` の penetration_5y — Pen(t)式の複製
3. メディア `mediaTechBenchmarkMetrics.ts` の m12_retention — `(1-churn)^12`(新規派生)

エンジン側の式が変わった場合、UIが黙って乖離する。エンジンがヘルパー関数をexportするか、keyMetricsに含める方が安全(→ D-9)。

### B-4. Python/TS実装間の潜在乖離: メディアのpaybackMonths【高】

- Pythonリファレンス: `paybackMonths` を **cpa > 0 のときのみ**出力(`if cpa > 0:` ブロック内)
- TSエンジン: churn > 0 なら **cpa=0でも** `paybackMonths: 0` を出力
- engine-spec §2.4 の文言は「ltvCpaRatio は cpa > 0 のときのみ」とだけ書かれており、paybackMonthsにはガードがない → **TS側がspec準拠、Python側が乖離**

golden fixtureのランダム生成はメディアの cpa ∈ [100, 5000] で cpa=0 を含まないため、keyMetricsキー集合比較(実装済み)がありながら未検出。さらに `arpuTotal=0` かつ churn>0 のとき paybackMonths=Infinity となり、JSON化やUI表示で問題になる(こちらは両実装共通の未定義挙動)。→ D-3

### B-5. プリセット「業界標準並/以下」の解釈のぶれ【中】

| プリセット | 説明文 | 実際の値 vs ダミー業界標準 | 整合 |
| --- | --- | --- | --- |
| SaaS②「利益率業界標準並」 | 営業利益率3% | 3% vs 3% | **完全一致** ✓ |
| SaaS③「利益率業界標準以下」 | -10% | -10% vs 3% | ✓ |
| EC②「F2転換率は業界標準並み」 | f2=22% | 22% vs **28%(-6pt)** | **不一致** ✗ |
| 創薬①「業界標準並みのフェーズ成功確率」 | 前3フェーズ65/52/29% | 完全一致 ✓ / ただしP3→承認=0.6×0.85=**51%** vs 58%(-7pt) | **部分不一致** △ |
| クライメート・メディア・医療機器 | 「業界標準」への明示的言及なし | ― | 対象外 |

SaaSは基準値に厳密に合わせているのに対し、EC②と創薬①は説明文と値がずれている。プリセットは仮値(U-17)だが、「説明文が業界標準に言及するなら値も一致させる」という運用ルールをセクター間で統一すべき。

### B-6. メディアフォームのラベル誤り【中】

`MediaTechForm.tsx` の `projectionYears` のラベルが「**DCF予測年数**」になっているが、メディアテックモデルにDCFは存在しない(EV/売上マルチプルのみ。projectionYearsは売上予測表示用)。誤解を招くラベル。

### B-7. ベンチマーク単位系の穴【低】

- `arpu_monthly_jpy` の unit が `"count"` — スキーマv1.1に「円単価」のunitが存在しない(`jpy_mn`はあるが単価用の`jpy`がない)ためのワークアラウンドで、Cowork側の実データ作成でも同じ問題に当たる
- `BenchmarkBar` のフォーマットは percent/x_multiple/ratio のみ単位サフィックス付きで、years/count は生の数値表示(「2.5」とだけ出て「年」が付かない)

### B-8. 編集可能リストの key={index}【低】

創薬の品目・マイルストーン、クライメートのCAPEX行がすべて配列indexをReact keyにしている。中間行の削除時にフォーム状態が行ズレするリスクのある既知のアンチパターン(現状はcontrolled inputのため実害は限定的)。

### B-9. エンジンのバリデーション網羅性の不均一【高寄りの中】

engine-spec §0.2は「ドメイン外入力は例外を投げず `ok: false` で返す(全域関数)」と定めるが、実装状況はセクターによって大きく異なる:

- SaaS: r ≤ g_term を検証 ✓
- 医療機器: recurringRatio ≥ 1、r ≤ g_term を検証 ✓
- **EC/メディア/クライメート/創薬: ほぼ無検証**

結果、フォームに `f2Rate=120%` を入力すると lifetimeYears = 1/(1-1.2) = -5 となり**負のLTVが正常値として表示**される。確率>100%、負の売上等も同様に素通りする。フォーム側にもmin/max属性がないため、ドメイン逸脱が実際に起こりうる。

## C. 要件が曖昧で独自判断した点(要決定)

各項目、**推奨**と代替案・トレードオフを付す。決定はユーザーに委ねる。

### C-1. 創薬のベンチマーク比較は先頭品目(assets[0])のみ

- 判断: ベンチマークは1品目単位の指標だがシナリオは複数品目を持てるため、先頭品目のみ比較(コメントで明記済み)
- 代替案: (a) 品目タブ切替で全品目比較 (b) POS加重平均で比較 (c) 現状維持
- **推奨: 現状維持、Phase 6で(a)を検討**。(b)は「フェーズ確率の平均」に意味論上の問題がある

### C-2. 算出定義が不明な4指標の比較除外

- 判断: `pipeline_value_ratio`(rNPV/何の比率か不明)、`capex_intensity`(CAPEX/何か不明)、医療機器`ev_sales_multiple`(モデルがマルチプル出力を持たない)、`paid_conversion`(対応ドライバーなし)を除外
- 代替案: (a) Cowork依頼票(cowork-benchmark-task.md)にmetric定義の明記を依頼しスキーマv1.2で確定 (b) エンジンにimplied指標(例: 医療機器のNTM売上からimplied EV/売上)を追加 (c) 現状維持
- **推奨: (a)を基本とし、医療機器のみ(b)も併用検討**。impliedマルチプルはDCF結果の妥当性チェックとしてVC実務でも使う指標のため価値が高い。(b)はエンジン+golden+Python参照実装の3点更新が必要

### C-3. 派生指標の定義式

- 判断: `m12_retention = (1-monthlyChurn)^12`、`penetration_5y = Pen(L+4)`、`pos_p3_to_approval = P(phase3)×P(filing)`
- 代替案: m12は「12ヶ月時点の残存率」ではなく「M12コホート継続率」等の別定義もありうる。ベンチマーク実データの定義次第で式を変える必要がある
- **推奨: Cowork依頼票に各metricの定義(分子/分母・時点)を明記してもらい、それに合わせて派生式を確定**。現状の式は報告書のこの節を根拠として仮確定扱いに

### C-4. Class III承認リードタイムの比較値に launchYear を使用

- 判断: `approval_lead_time_class3`(2.5年)と `launchYear`(現在からの上市年数)を比較
- 問題: launchYearには「現在のフェーズから承認まで」の年数が入るため、開発が進んでいる品目ほど短くなり「承認リードタイム」の比較として意味がずれる
- 代替案: currentPhaseから申請完了までの累積duration(エンジンのcumThroughと同じ計算)を使う
- **推奨: 代替案へ変更**(自己指摘。現実装は簡便だが意味論が弱い)

### C-5. EC の ev_sales_multiple 比較は multipleBasis='revenue' のときのみ

- 判断: 粗利基準のマルチプルと売上基準の業界標準を混ぜて比較しない
- 代替案: 粗利基準時に `evMultiple.base × grossMargin` で売上換算して比較
- **推奨: 現状維持**。換算表示は「入力値と違う数字が出る」混乱を招く

### C-6. マイルストーンのフェーズ選択肢に現フェーズより前を含む

- 判断: UI上は全フェーズ+上市時を選択可。エンジンは過去フェーズ指定を「t=0・確率1で計上」(エンジン側にTODOコメントあり)
- 代替案: UIの選択肢を「残フェーズ+上市時」に制限
- **推奨: UI制限(低コストで意味論の穴を塞げる)**。過去フェーズのマイルストーン(既受領分)を記録したい場合は将来「受領済みマイルストーン」として別扱いにする

### C-7. プリセット適用は即保存

- 判断: プリセットボタン押下で draft を差し替えて即 onSave(未保存の編集は確認なしで消える)
- 代替案: 適用はdraftのみ、保存は手動ボタンに統一
- **推奨: 代替案(適用≠保存)**。現状はUI一貫性(他の編集は保存ボタン必須)と食い違う

### C-8. metric_id契約の解消方法

- 判断: UIマッピング表で橋渡し(現状)
- 代替案: (a) スキーマのdescription修正「UI層のマッピング表(`{sector}BenchmarkMetrics.ts`)で対応付ける」 (b) エンジンドライバーをsnake_caseに改名
- **推奨: (a)**。(b)はエンジン・golden・Python参照実装・テスト全てに波及しリスクに見合わない

### C-9. 「主要品目フェーズ失敗」の表現

- 判断: Phase2成功確率5%(失敗イベントそのものではなく「ほぼ確実な失敗の見込み」)
- 代替案: 確率0%(完全失敗。売上寄与ゼロ+投下済み開発費で負のrNPVになる)
- **推奨: 現状維持だが、プリセット説明文に「確率5%に低下」と数値を明記**。0%はエンジンの境界値としては正しく動く(golden検証済み)ので、UIから選べる「完全失敗」プリセットを追加する選択肢もある

## D. 修正提案(優先度順)

### 高(計算の正しさ・データ破壊)

- **D-1. 旧シナリオデータでシナリオ詳細ページがクラッシュする**
  Phase 3で `Scenario` 型に `vcMethod` を追加したが、localStorageのキーは `scenarios:v1` のまま**マイグレーション処理がない**。Phase 2時点で作成・保存されたシナリオ、またはPhase 2形式でエクスポートされたJSONをインポートした場合、`scenario.vcMethod` が undefined となり、`VcMethodSection` の `vcMethod.netDebtAtExit` アクセスで TypeError → 詳細ページがクラッシュする。ロード時・インポート時にデフォルト値を補完するマイグレーションを追加すべき(§8「localStorage消失対策」の趣旨にも関わる)。
- **D-2. エンジン入力バリデーションの網羅(B-9)**
  spec §0.2の全域関数原則に沿い、全セクターでドメイン検証を実装し `ok: false` を返す。優先: EC `f2Rate ∈ [0,1)`、確率系 `[0,1]`、メディア `arpuTotal=0 ∧ churn>0` 時のInfinity対策。フォーム側のmin/max属性も併せて。
- **D-3. メディア paybackMonths のPython/TS乖離解消(B-4)**
  Python参照実装をspec準拠(cpa>0ガードを外す)に修正し、golden境界ケース(cpa=0・churn>0)を追加して再生成。TS側は現状維持。あわせてspec §2.4にpaybackMonthsのガード有無を明記。

### 中(整合性)

- **D-4.** Comps企業の出典・取得日表示(A-1)
- **D-5.** ダミーバッジをヘッダー等にアプリレベル常設(A-2)
- **D-6.** 創薬の感度分析ドライバー定義追加(B-1。Phase 4の前提条件)
- **D-7.** EC②・創薬①プリセットの説明文と値の整合(B-5。値を業界標準に合わせるか説明文を「やや下回る」に変える)
- **D-8.** メディアフォーム「DCF予測年数」→「売上予測年数」(B-6)
- **D-9.** 派生指標(POS・penetration_5y・m12)のエンジン側ヘルパー化(B-3)
- **D-10.** metric_id契約のスキーマ文言修正(A-7/C-8)
- **D-11.** 並列比較ビュー(A-4)・金額単位切替(A-5)・期待IRR/MOIC表示(A-3)のフェーズ割当決定

### 低(改善)

- **D-12.** 編集可能リストのkeyをindexからid(uuid)に(B-8)
- **D-13.** `basis`・`notes`・`as_of` の表示(A-6)
- **D-14.** years/count単位のサフィックス表示(B-7)
- **D-15.** `EvRangeResult` の auxiliary ラベル「簡易DCF」ハードコード(現状SaaSのみのため実害なし)
- **D-16.** エンジンが返す `cashflows`(チャート用)が全ビュー未使用。Phase 6のチャート実装で使う予定なら仕様として明記、使わないなら削除
