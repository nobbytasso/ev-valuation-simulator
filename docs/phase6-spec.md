## 1. T1: トークン体系の第2層拡張とコンポーネントCSSの棚卸し

テーマの「人格」を作る装飾は、すべて**トークンとして2セット定義**し、コンポーネントCSSはトークン参照のみとする(設計原則3。`if (theme === ...)` 禁止は据え置き)。

追加するトークン(両テーマで値が異なる第2層):

```
--pattern-bg                  背景パターン(ダーク: 微細グリッド+スキャンライン(CSS gradient)、ライト: 淡いドット)
--panel-border-style          パネル罫(ダーク: 1pxヘアライン+コーナーマーカー用変数、ライト: なし)
--corner-marker-size / -color コーナーマーカー(ダークのみ視認、ライトは transparent)
--shadow-glow-strong          数値強調用グロー第2段(ライト: none)
--surface-overlay             半透明レイヤー(ダーク: 電脳オーバーレイ感、ライト: 白の淡い重ね)
--caption-font / -color       日英併記キャプション用(§6.3)
--gauge-track / -fill / -lock 円形ゲージ配色(§4)
--motion-duration-slow        演出用(スキャン走査・パーティクル寿命)
--focus-ring                  フォーカス可視化(AA対応の一環)
```

- 併せて全11個のコンポーネントCSSを棚卸しし、**ハードコードされた色・影・角丸をトークン参照に置換**する(検査方法: CSS内の `#`/`rgba(` 直書きを grep し、意図的な例外はコメントで根拠を書く)。
- パネルの見た目(ダーク: 細線ワイヤーフレーム+コーナーマーカー+塗りより線 / ライト: ふんわりソフトシャドウ+大きめ角丸+リボン風バッジ)は共通クラス(例: `.panel`)に集約し、各セクションの個別CSSから重複を除く。
- **レイアウト(グリッド構造・配置・box寸法)は両テーマ共通**を維持。テーマ切替でレイアウトシフトが起きないことを E2E で検証する(§8.2)。

## 2. T2: Webフォント導入(§8 非機能)【P6-3裁定: Google Fonts CDN 採用】

- **Google Fonts CDN を利用する**(2026-07-14 裁定で代替案を採用。`docs/logs/phase6-rulings-20260714.md`)。トレードオフ(利用者IPが Google に渡る・外部可用性依存)はユーザー了承済み。設計原則4「入力データはブラウザ外に出さない」には抵触しない(フォント取得はアプリの入力値を一切送信しない)が、**Stage 1 で唯一の外部リクエスト**となるため、その旨を裁定記録に残した。
  - ダーク: Inter(見出し/本文。400/700)+ JetBrains Mono(等幅数値・英字キャプション。400/700)
  - ライト: M PLUS Rounded 1c(見出し。400/700)。日本語グリフは Google Fonts の **unicode-range 自動サブセット配信**に委ねる(必要グリフ分のみ取得され、初期ロード<3秒制約と両立)
  - 本文の日本語は引き続きシステムフォント(Hiragino)を第一候補とし、Webフォントは見出し・数値・キャプションに限定する(転送量と<3秒制約のため)
- `index.html` の `<head>` に `<link rel="preconnect" href="https://fonts.googleapis.com">` と `<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>`、css2 API の `<link rel="stylesheet">`(**`display=swap` パラメータ必須** = font-display: swap)を追加して FOUT を最小化する。npm 依存の追加はなし。
- **フォールバック必須**: CDN 到達不能(オフライン・遮断環境)でもトークンのフォールバックチェーン(Hiragino / system-ui / ui-monospace)で全画面が機能・判読を保つこと(E2E はフォント取得の成否に依存させない)。
- 数値表示は両テーマとも `font-variant-numeric: tabular-nums`(既存 base.css の `.tabular-numbers` を主要数値コンポーネントに適用徹底)。

## 3. T3: 金額表示単位切替(百万円/億円)【Rev.5割当】

### 3.1 共通フォーマッタと単位ストア

```ts
type MoneyUnit = 'million_yen' | 'oku_yen'   // 百万円 / 億円(1億円 = 100百万円)

// src/ui/format/money.ts(純粋関数)
function formatMoney(valueInMillionYen: number, unit: MoneyUnit): string
function moneyUnitLabel(unit: MoneyUnit): string          // '百万円' | '億円'
function moneyAxisLabel(unit: MoneyUnit): string          // チャート軸・表ヘッダ用

// src/ui/format/useMoneyUnit.ts — テーマ選択と同じパターン(localStorage永続、ヘッダのトグルで切替)
```

- 表示桁(P6-2 推奨): 百万円 = 整数(現行どおり)、億円 = **小数1桁**(例: 10,400百万円 → 104.0億円。1億円未満は 0.5億円のように読める)。
- 6ファイルに重複する `formatMoney` と各所のインライン「百万円」表記を **`src/ui/format/` に一本化**する(重複の解消自体が T3 の前提作業)。表ヘッダ・チャート軸ラベル・キャプションの「(百万円)」も `moneyUnitLabel` 参照に置換。
- ヘッダに単位トグル(テーマトグルと並ぶ)。選択は localStorage(`ev-valuation-simulator:money-unit`)に永続化、既定 = 百万円。

### 3.2 適用範囲(P6-1)

- **推奨: 表示系のみ切替**。結果ビュー(EVレンジ・VC法・資本政策・感度分析)・比較ビュー・ポートフォリオ・チャート軸。
- **切替対象外(百万円固定)**: ①入力フォーム(入力値の意味が変わり誤入力を誘発するため。フォームラベルの「(百万円)」は固定のまま) ②**Excel出力**(§5 制約4「プレーン表現」+ Phase 5 検収済みのヘッダ契約を維持。単位はヘッダ明記済み) ③エンジン・永続化(Money 型は百万円のまま。変換は表示直前のみ)。

## 4. T4: 円形ゲージ(主要数値のメーター表示)

### 4.1 コンポーネント(共通・トークン参照のみ)

```ts
// src/ui/gauge/CircularGauge.tsx
interface CircularGaugeProps {
  label: string                 // 例: '企業価値(ベース)'
  captionEn?: string            // 英字キャプション(§6.3。例: 'ENTERPRISE VALUE')
  value: number                 // 表示値(フォーマット済み文字列は valueText で渡す)
  valueText: string             // 中央のタブラー数字(単位込み)
  ratio: number | null          // リング充足率 [0,1]。null はリングを不定様式(トラックのみ)で描画
  status?: 'good' | 'caution' | 'bad' | 'neutral'   // 判定色(§5)。既定 neutral
}
```

- SVG 同心円リング(トラック+フィル)。色・線幅・グローはすべてトークン(`--gauge-*`)。ライトでは同一ジオメトリのままパステル配色+ソフトな見た目(レイアウト共通の原則)。
- 回転アニメーション・確定演出(リング収束→数値ロック)は **theme-effects 側**(§6)から CSS クラスを付与する形にし、ゲージ本体はアニメなしでも完結する(reduced-motion 時は静的表示)。
- 数値のカウントアップ(両テーマ共通モーション)は framer-motion の `animate` を用いた共通フック `useCountUp(value)`(reduced-motion 尊重、初回のみ)。

### 4.2 適用箇所(P6-4)と ratio の定義

推奨: **結果セクションの主要値に限定**(過剰適用はチャート・数表の判読を妨げる):

| 値 | ratio(リング充足率) |
| --- | --- |
| EVベース(EvRangeResult) | (base − pessimistic) / (optimistic − pessimistic)。レンジ縮退時は null |
| Rule of 40(SaaSのみ) | min(max(value/80, 0), 1)。**業界標準線(40)をリング上のマーカーで表示** |
| VC法 含意IRR | min(irr / 1.0, 1)(100%を上限とする表示スケール。意味付けではなく表示上の正規化) |
| 期待IRR / 期待MOIC(資本政策) | IRR: 同上 / MOIC: min(moic / 10, 1)(目標倍率10xを基準)。算出不能時はゲージを出さず従来の「—(理由)」表示 |

ratio はあくまで**表示上の正規化**であり評価判定ではない(判定は §5 の判定色が担う)。ratio の正規化定数(80 / 1.0 / 10)はマジックナンバーとしてトークンではなく定数モジュールに置き、根拠コメントを付す。

## 5. T5: 判定色システム(段階状態色)

### 5.1 意味論

状態は3段階+中立: `good`(良好: ダーク=シアン/ライト=グリーン系) / `caution`(注意: アンバー) / `bad`(悪化: マゼンタ〜レッド) / `neutral`(判定なし)。トークンは既存の `--color-status-*` を使う(値の微調整は T1)。

判定を出す場所と規則(**新しい閾値を発明しない** — 既存の業界標準値・既存フラグのみを根拠とする):

| 適用箇所 | 規則 |
| --- | --- |
| ベンチマーク対比(BenchmarkBar の差分表示) | 業界標準との差 × 指標の極性(§5.2)。極性 neutral の指標は色を付けない |
| VC法「成立不可」フラグ | bad(既存 isInfeasible) |
| EV ≤ 0 の警告(創薬・クライメート) | bad(既存警告の色格上げ) |
| 資本政策・ポートフォリオの「—(理由)」 | caution(算出不能は悪化ではなく注意) |
| 円形ゲージ(§4) | 上記判定を status プロップで受ける(ゲージ自身は判定しない) |

### 5.2 指標の極性表(P6-5。**ドラフト — 裁定対象**)

`BenchmarkMetricConfig` に `direction: 'higher_better' | 'lower_better' | 'neutral'` を追加する。ドラフト(全25 metric):

| direction | metric_id |
| --- | --- |
| higher_better | arr_growth_yoy, revenue_growth_yoy, gross_margin, operating_margin, nrr, rule_of_40, ltv_cac, m12_retention, f2_conversion, pos_preclinical_to_p1, pos_p1_to_p2, pos_p2_to_p3, pos_p3_to_approval, penetration_5y, recurring_revenue_ratio, offtake_coverage, scaleup_success_prob, royalty_rate(導出側の受取率), arpu_monthly_jpy |
| lower_better | ad_cost_ratio, subsidy_dependency, approval_lead_time_class3 |
| neutral(色なし) | ev_arr_multiple, ev_sales_multiple(高い=割高/強気であり良し悪しではない), discount_rate(リスク認識の入力) |

caution の帯域: 差分が業界標準の **±10%以内は caution(標準並み)**、それを超えて好方向なら good・悪方向なら bad、を推奨(帯域幅も裁定対象。代替: 帯域なしの2値)。

## 6. T6: テーマ固有演出(theme-effects/)

すべて `src/theme-effects/` に隔離し、コンポーネント本体はフック/クラス名を受けるだけにする。**共通の制約**: 初回のみ・短時間(≤600ms 目安)・`prefers-reduced-motion` で完全無効(CSS は既存ガードが効く。JS パーティクルは `matchMedia` で発火自体を止める)・チャートと数表の上に被せない。

### 6.1 ダーク(FUI)

| 演出 | 実装方針 |
| --- | --- |
| 背景の微細グリッド+スキャンライン | 静的 CSS(`--pattern-bg`、repeating-linear-gradient。淡い不透明度で判読を妨げない) |
| 計算実行時のスキャン走査 | 結果パネルを1回走査する擬似要素アニメ(600ms)。トリガーは P6-8(推奨: プリセット適用時とシナリオ切替時のみ。キー入力毎は §5 制約2「操作を待たせない」に反するため発火しない) |
| パネル出現時のタイプイン/デコード風表示 | 見出しテキストに一度だけ適用する軽量CSS/フック(初回マウント時のみ) |
| 円形ゲージの回転+確定演出 | §4 のクラス付与(リング描画→収束→数値ロック。カウントアップと同期) |

### 6.2 ライト(ギャルゲー風ポップ)

| 演出 | 実装方針 |
| --- | --- |
| ハート/スパークルのパーティクル | 保存成功・プリセット適用時に短時間(≤600ms・8〜12粒)。`theme-effects/particles.ts` + portal。データ表示領域には重ねない |
| ボタンのぷにスケール | CSS `:active` scale(トークン化したイージング)。テーマ共通CSSに置き、ダークでは控えめな値になるようトークンで差を付ける |
| 背景の淡いドット/ストライプ | 静的 CSS(`--pattern-bg`) |

### 6.3 日英併記キャプション(ダークの様式だが DOM は両テーマ共通)

- 主要見出しに英字キャプションを付す(例: 「企業価値 / ENTERPRISE VALUE」「感度分析 / SENSITIVITY」)。**DOM には両テーマとも常設**し、スタイルのみテーマで変える(ダーク: 等幅小サイズ大文字シアン / ライト: 小さな欧文・ピンク系)。表示/非表示をテーマで切り替えるとレイアウトシフト禁止則に抵触するため(P6-6 で裁定)。
- 対象はセクション見出し(結果 / VC法 / 感度分析 / 資本政策 / ベンチマーク比較 / 比較 / ポートフォリオ)+円形ゲージのラベル。英字文言一覧を `src/ui/sectionCaptions.ts` に定数表として定義(発明ではなく一般的な財務用語の英訳。実装時に一覧をレビュー可能にする)。
- **IP上の注意**(要件 §5): 作中ロゴ・キャラクター・固有名詞・フォントの複製はしない。様式(細線+シアン発光+円形ゲージ+段階判定色)の参照に留める。

## 8. T8: 品質ゲートとデザインレビュー

### 8.1 自動検証(ユニット)

- **WCAG AA コントラストテスト**: トークンCSSをパースし、相対輝度→コントラスト比を計算する純粋関数で、両テーマの主要ペア(text/bg・text-muted/bg・accent-contrast/accent・status各色/bg・キャプション色/bg)が **4.5:1 以上**(大きな数値表示は 3:1 以上)であることをテストで固定する。パステル地の色調整はこのテストを Green に保ったまま行う(要件 §5 制約1 の機械化)。
- 単位切替フォーマッタの境界(0 / 負値 / 100百万円=1.0億円 / 丸め)。

### 8.2 E2E(既存11本の維持+追加)

1. テーマ切替の**レイアウトシフトなし**: 代表画面で主要要素の boundingBox をダーク/ライトで比較(±1px)。
2. reduced-motion: パーティクル・スキャン走査・カウントアップが発火しない(DOM に演出要素が現れない)。
3. 単位トグル: 億円表示→リロードで永続→表とチャート軸ラベルが「億円」になる。Excel は百万円のまま(ダウンロード読み戻し)。
4. **既存テスト8(FOUC)の期待値更新に注意**: トークン値を変更する場合、`--color-bg`/`--color-text` の検証値を同一コミットで更新する。

### 8.3 デザインレビューの進め方(完了条件の運用。P6-9)
