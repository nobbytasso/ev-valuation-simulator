# Coworkプロジェクト指示書: ベンチマークデータ収集・JSON化(v1.2対応)

> このファイルを Claude Cowork の別プロジェクトに投入して使用する。シミュレーター本体の開発とは独立して随時実行し、成果物JSONを本体リポジトリの `data/benchmarks/` に差し替える。
>
> **v1.1からの改訂(2026-07-13、一括改訂)**: ①各metricに `definition`(指標の定義: 分子/分母/基準時点)の記入を**必須化** ②円建て単価の unit に `"jpy"` を新設(従来 `count` で代用していた箇所を移行) ③`metric_id` の契約を明確化(下記)。**旧v1.1形式での納品は受け入れない。**

## 目的

VC向け企業価値シミュレーターで使用するベンチマークを公開資料から収集し、`benchmark.schema.json`(**v1.2**)準拠のJSONとして納品する。

## 方針(重要)

分布(四分位)方式は廃止。収集するのは以下の2種類のみ:

1. **業界標準値**(`reference_type: "industry_standard"`): 指標ごとに単一の基準値。公開レポートや業界慣行に基づく代表値でよい(1指標1件)
2. **比較対象企業**(`reference_type: "comp_company"`): セクターごとに代表的な上場Compsを**1〜3社**選定し、主要指標の実績値をIR資料から転記(企業名を `company_name` に記入)

大量のCompsを集計する必要はない。少数精鋭のCompsと信頼できる標準値で足りる。

## v1.2の必須事項

### definition(指標の定義)— 全metricで必須

各エントリの `definition` フィールドに、**分子/分母/基準時点**が特定できる定義を1〜2文で記入する。本体UIはこの定義に合わせてシナリオ入力値からの換算式を確定するため、定義のない数値は比較に使えない。

例:
- `m12_retention`: 「登録12ヶ月後に残存しているユーザー数 ÷ 12ヶ月前の新規登録ユーザー数。月次コホートベース」
- `penetration_5y`: 「上市5年目の年間使用手技数 ÷ 同年の対象手技総数」
- `pipeline_value_ratio`: 「(定義を発見した資料の通りに記載。時価総額比か、ピーク売上比か等を明記)」
- `capex_intensity`: 「累計CAPEX ÷ 年間売上、量産到達時点」等、**何と何の比率かを必ず明記**

※ v1.1で納品定義が曖昧だった `pipeline_value_ratio` / `capex_intensity` / `paid_conversion` は、definitionが明確にならない限り本体UIでは比較対象外として扱われる。

### unit の使い分け

| unit | 用途 |
|---|---|
| `jpy` | **円建て単価**(ARPU、平均注文単価、手技あたり単価等)【v1.2新設】 |
| `jpy_mn` | 百万円(企業規模の金額) |
| `percent` / `ratio` / `x_multiple` / `months` / `years` / `count` | 従来通り(`count` は人数・件数等の無単位数のみ。単価に使わない) |

### metric_id の契約

`metric_id` は本体UI層のマッピング表(`src/ui/sectors/*/…BenchmarkMetrics.ts`)が対応付けるキー(snake_case)。エンジンのドライバーIDと直接一致はしない。**既存のmetric_id(下表)は変更しないこと。新規metricを追加したい場合は本体プロジェクトに連絡し、マッピング表の登録とセットで行う。**

## 作業手順

1. `benchmark.schema.json`(**v1.2**)を読み、出力フォーマットを厳守する(`schema_version: "1.2"`)
2. セクター1つずつ、公開情報をWeb調査。**一次情報(IR資料・公的資料)を優先**
3. 業界標準値は根拠を `basis` に短く記載(例: 「○○レポート2026年版の中央値」「業界慣行」)。**あわせて `definition` に指標の定義(分子/分母/基準時点)を必ず記載**
4. Compsは各社のIR・決算資料から指標値を転記し、`basis` に会計期間を明記(例: 「FY2025通期実績」)。`definition` は業界標準値側と同一定義であることを確認(異なる場合はその旨をnotesに)
5. 収集ログ(企業/資料/取得値)をスプレッドシートに整理(将来のGAS配信の元データを兼ねる)
6. `data_status: "production"` でJSONを出力し、スキーマバリデーションを実行して納品

## 制約・注意

- **出典のない数値は納品しない**(不明な指標は欠落させてよい。推定で埋めない)
- **定義(definition)のない数値も納品しない**(定義が資料から特定できない場合は欠落させ、収集ログに理由を残す)
- 有料レポートの数値の転載は行わず、引用可否が不明な資料は `license_note` に懸念を記録して人間の判断を仰ぐ
- 基準日 `as_of` と取得日 `retrieved_at` を混同しない
- 更新時は既存JSONを上書きせず、`as_of` 別にファイルを残す(履歴管理)
- Comps企業の選定理由を収集ログに1行で残す

## 納品物

1. `benchmarks_{sector}_{as_of}.json` × 6セクター(schema v1.2、全metricにdefinition付き)
2. 収集ログのスプレッドシート(出典一覧・Comps選定理由・definition根拠)
3. 前回版との差分サマリ(2回目以降)
