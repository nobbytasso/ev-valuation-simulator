# VC向け企業価値シミュレーター 要件定義書 Rev.4

Rev.3 からの変更点: ダークモードのデザインベースを「攻殻機動隊S.A.C.」「PSYCHO-PASS」のUI様式を参考とする方向に具体化(雰囲気・様式の参照であり、作中素材の複製は行わない)。 Claude Code に投入する自己完結型仕様。

---

## 1\. 目的と利用者

- 利用者: ベンチャーキャピタリスト(および投資検討チーム)  
- 目的: セクター別キードライバーに基づく企業価値のシナリオ別シミュレーションと、業界標準値・比較対象企業(Comps)との対比  
- 言語: 日本語UI(金額は百万円単位を基本、切替可)

## 2\. アーキテクチャ(2段階構成)

### Stage 1 — MVP(GitHub Pages / バックエンドなし)

- ホスティング: GitHub Pages(静的)  
- シナリオ・ポートフォリオ保存: localStorage \+ JSONインポート/エクスポート  
- ベンチマーク: リポジトリ内 `data/benchmarks/` の静的JSON。**Stage 1はダミーデータ**(`data_status: "dummy"` 必須、UIに「ダミーデータ」バッジ常時表示)  
- 実データJSONは別プロジェクト(Claude Cowork)で作成し、同一スキーマでファイル差し替えのみで反映

### Stage 2 — 本格利用(Google Cloud 前提)

| 用途 | 推奨 | 理由 |
| :---- | :---- | :---- |
| ベンチマーク配信・更新 | GAS \+ Googleスプレッドシート | 担当者がシート編集 → GASウェブアプリがJSON配信。見直し運用と合致。CacheServiceで配信キャッシュ |
| シナリオ保存/共有 | Google Drive API \+ Google Identity Services(OAuth) | クライアントサイドOAuthのみで、静的ホスティングのままユーザー別保存・端末間同期・チーム共有が可能 |
| 将来(同時編集・権限・監査) | Cloud Run \+ Firestore へ移行 | スコープ外。アダプタ設計で移行容易性を確保 |

### 移行容易性の要(必須設計)

  list / load / save / delete / export / import

}

  fetchSector(sectorId): BenchmarkData

}

UI・計算エンジンはインターフェースのみに依存。GAS/Drive SDK呼び出しはアダプタ内に閉じる。

### 技術スタック(Stage 1\)

React \+ TypeScript \+ Vite / Zustand / Recharts / Framer Motion / SheetJS / fast-check

### 設計制約

1. 計算エンジン `src/engine/` は依存ゼロの純粋TypeScript。UI・ストレージから完全分離  
2. 入力データはブラウザ外に送信しない(Stage 2のDrive/GAS送信はユーザー明示操作のみ)

## 3\. セクター定義とキードライバー(6セクター)

### 3.1 SaaS(日本市場)

- 評価手法: EV/ARRマルチプル(主)+ 簡易DCF(補助)  
- ドライバー: ARR、ARR成長率、NRR、グロスマージン、営業利益率/FCFマージン、Rule of 40(自動算出)、チャーン、CAC回収期間  
- シナリオプリセット: ①順調(Rule of 40達成継続) ②成長鈍化・利益率業界標準並 ③停滞・利益率業界標準以下

### 3.2 創薬

- 評価手法: rNPV  
- ドライバー: パイプライン(複数品目)、フェーズ、フェーズ別成功確率(既定値同梱・編集可)、上市年、ピーク売上、売上カーブ、自社販売/導出(ロイヤリティ・マイルストーン)、割引率(既定10–12%)  
- シナリオ: ベース/導出成功/主要品目フェーズ失敗

### 3.3 医療機器

- 評価手法: 市場浸透モデル \+ DCF(またはEV/売上)  
- ドライバー: 対象患者数・手技数、クラス分類、承認タイムライン(PMDA/FDA)、保険償還、浸透率カーブ、リカーリング比率、販売チャネル  
- シナリオ: 承認遅延/償還下振れ/海外展開加速

### 3.4 メディアテック

- 評価手法: EV/売上 \+ ユーザーエコノミクス  
- ドライバー: MAU/DAU、ARPU(広告/課金/コマース構成)、継続率カーブ、コンテンツ原価率、CPA  
- シナリオ: 広告市況悪化/課金転換率改善/ヒット有無

### 3.5 EC/D2C

- 評価手法: EV/売上(またはEV/粗利)+ ユニットエコノミクス  
- ドライバー: 年間売上/GMV、成長率、粗利率、F2転換率、LTV/CAC、平均注文単価、購入頻度、在庫回転、広告費率、物流費率  
- シナリオ: ①リピート育成成功 ②広告依存継続(CAC上昇) ③成長鈍化・在庫リスク

### 3.6 クライメートテック

- 評価手法: プロジェクトDCF(CAPEX重視)+ マイルストーン到達確率によるリスク調整  
- ドライバー: CAPEX、量産化到達確率、オフテイク契約カバー率、補助金依存度、ユニットコスト低減カーブ、カーボンクレジット価格感応度  
- シナリオ: ①制度追い風+オフテイク確保 ②補助金縮小 ③量産化遅延

### 共通オーバーレイ

- VC法(Exit企業価値 ÷ 目標倍率)、Exit想定(IPO類似マルチプル / M\&A想定倍率)  
- 出力: 企業価値レンジ(悲観/ベース/楽観)、IRR / MOIC

## 4\. 機能要件

### 4.1 必須

1. **シナリオ管理**: 作成・保存・複製・命名・削除、並列比較ビュー(表+チャート)  
2. **ベンチマーク比較(簡易方式)**【Rev.2で変更】:  
   - 分布(四分位)表示は**廃止**。Compsを十分な数集めることが現実的でないため  
   - 比較対象は2種類: **業界標準値**(指標ごとの単一基準値、例: Rule of 40 \= 40)と**比較対象企業**(実在Compsを1〜数社、企業名+指標値)  
   - UI: 入力値・出力値のゲージ/バーチャートに基準線(マーカーライン)として重畳表示。「業界標準比 \+12pt」のような差分表示  
   - 全基準値に出典・取得日を表示。ダミーデータ時はバッジ表示  
3. **仮想ポートフォリオ管理**: 複数企業の投資額・ラウンド・持分登録、ファンド単位集計(IRR/MOIC/時価総額)  
4. **Excelエクスポート**: シナリオ単票/比較表/ポートフォリオサマリ(値+前提条件シート)  
5. **感度分析(トルネードチャート)**: ドライバー±変動時の企業価値インパクトを自動算出・降順表示(既定±20%、変更可)  
6. **資本政策・希薄化シミュレーター**: 将来ラウンド(調達額・プレバリュー・オプションプール)を時系列に積み、持分推移とExit時の手取り・IRR/MOICを計算。シナリオの評価結果と連動  
7. **デザイン**【Rev.2で変更】: 後述「5. デザイン要件」参照。ダーク/ライトモード切替必須

### 4.2 任意(将来)

モンテカルロ / シナリオ共有URL / 前提条件の監査ログ / スナップショット履歴

## 5\. デザイン要件【Rev.3で改定: デュアルテーマ構成】

### 方向性

金融ダッシュボード調は不要。ダークとライトで**まったく異なる人格**を持つデュアルテーマとする:

- **ダークモード(既定)= FUI**: 映画に登場するインターフェース(Fictional UI)を思わせる最先端感。参考イメージ: SF映画のホログラフィックHUD、ミッションコントロール、サイバーパンク系コンソール  
- **ライトモード \= ギャルゲー風ポップ**【Rev.3で変更】: 白+パステル調、アクセントはラブリーなピンク。恋愛シミュレーションゲームのメニュー/ステータス画面のような、かわいく親しみやすい画面

### ダークモード(FUI)の具体要素【Rev.4で具体化】

**デザインベース: 「攻殻機動隊 S.A.C.」および「PSYCHO-PASS」のUI様式を参考とする。** ただし雰囲気・様式の翻案に留め、作中のロゴ・キャラクター・固有UI画面の複製や模写は行わない(下記「IP上の注意」参照)。

- **配色(S.A.C.系)**: ほぼ黒に近い深いグリーン〜ネイビーの地に、シアン/ティール系の発光ライン。警告・強調にアンバー/オレンジを限定使用(例: 業界標準を下回る指標、リスクシナリオ)  
- **線とレイヤー(S.A.C.系)**: 細いヘアライン罫のワイヤーフレーム風パネル、コーナーマーカー、半透明レイヤーの重なり(電脳空間のオーバーレイ感)。塗りより線で構成する  
- **円形ゲージ(PSYCHO-PASS系)**: 企業価値・IRR/MOIC・Rule of 40等の主要数値は、同心円リング+回転アニメーションの円形メーターで表示。数値は大きなタブラー数字を中央に置き、計測完了時に確定演出(リングが収束して数値がロックされる)  
- **判定色(PSYCHO-PASS系)**: シナリオ判定・ベンチマーク対比の状態色を段階制にする — 良好: シアン系 / 注意: アンバー / 悪化: マゼンタ〜レッド系(色相の段階で状態が直感的に読める)  
- **テキスト演出**: パネル出現時の一瞬のタイプイン/デコード風表示、ラベルは日本語+英字キャプション併記(例: 「企業価値 / ENTERPRISE VALUE」)、等幅英字はテクニカルな印象のフォントを使用  
- **背景**: 微細なグリッド/スキャンライン、ごく淡いノイズ。計算実行時の短いスキャンライン走査演出  
- **質感**: グロー(box-shadow/フィルタ)は発光ラインと数値に限定し、面には使わない。シャープな角(角丸小)

**IP上の注意(必須)**: タチコマ等のキャラクター、笑い男マーク、公安9課/シビュラシステムのロゴ・エンブレム、ドミネーターUIの直接的な再現、作中の固有名詞・フォントの複製は行わない。あくまで「細線ワイヤーフレーム+シアン発光+円形ゲージ+段階判定色」という様式の参照とする。

### ライトモード(ギャルゲー風ポップ)の具体要素

- **配色**: ホワイト基調+パステル(ピンク/ラベンダー/ミント/クリームイエロー)、アクセントカラーはラブリーピンク(例: \#FF8FC7 系。実装時にトークンで調整)。グラデーションは淡く  
- **形状**: 大きめの角丸(パネル・ボタンとも)、ふんわりしたソフトシャドウ、ピル型ボタン、リボン/バッジ風のラベル  
- **装飾モチーフ**: ハート・キラキラ(スパークル)・ドット/ストライプの淡い背景パターンを**控えめに**使用(データ表示領域には重ねない)  
- **タイポグラフィ**: 見出しは丸ゴシック系(例: M PLUS Rounded 1c / Zen Maru Gothic)、本文は可読性優先の標準ゴシック  
- **モーション**: ボタンのぷにっとしたスケール、ハート/スパークルの短いパーティクル(計算完了時など)、ゆるいイージング  
- 切替はヘッダーのトグルで即時反映、選択をlocalStorageに永続化

### 共通モーション

画面遷移のスライド/フェード、数値のカウントアップ、チャートのドローアニメーション、ホバー時のマイクロインタラクション。Framer Motion \+ CSS。数値表示は両テーマともタブラー数字の等幅系

### 制約(演出より優先)

1. 数値の可読性・コントラスト比(WCAG AA相当)を両テーマで犠牲にしない(パステル地に白文字などのAA割れを禁止。ピンクアクセントの文字使用時は濃色シェードを用いる)  
2. アニメーションは初回のみ/短時間とし、操作を待たせない。`prefers-reduced-motion` を尊重  
3. 装飾モチーフ(グリッド/スキャンライン、ハート/スパークル)はチャート・数表の判読を妨げない不透明度に抑える  
4. Excel出力・印刷系のアウトプットは両テーマ共通で装飾を排したプレーン表現とする

### 実装上の指示(デュアルテーマの要)

- テーマトークンは**色だけでなく、角丸(radius)・影(shadow)・フォントファミリー・背景パターン・モーションプリセットまで含めて**2セット定義する(テーマ切替で人格が変わるため、色トークンのみでは成立しない)  
- コンポーネントはトークン参照のみとし、テーマ条件分岐をコンポーネント内に書かない(例外: 装飾パーティクル等のテーマ固有演出は `theme-effects/` に隔離)  
- レイアウト(グリッド構造・要素配置)は両テーマ共通とし、切替時のレイアウトシフトを起こさない  
- Claude Code の Phase 6 着手時に frontend-design スキル(利用可能な場合)を参照する

## 6\. ベンチマークデータ仕様【Rev.2で改定】

- 作成は別プロジェクト(Claude Cowork)。本リポジトリはスキーマ準拠JSONの取り込みのみ  
- スキーマ: `benchmark.schema.json`(**v1.1: 分布→単一基準値方式に簡素化**)  
  - 各エントリ: `metric_id` / `reference_type`("industry\_standard" | "comp\_company") / `company_name`(comp時のみ) / `value` / `source`  
  - 1指標につき業界標準1件+比較企業0〜3社程度を想定  
- Stage 1: `benchmarks.dummy.json`(同梱)を使用  
- 差し替え: 同スキーマのJSONを置き換えるだけで反映されること(受入条件)  
- Stage 2: GASエンドポイントから同一スキーマで配信

## 7\. Phase 1 検証データの自動生成

1. **リファレンス実装との突合**: Pythonで各セクターモデルを独立実装(`tools/reference/`)、シード固定で入力を多数生成し期待出力を `engine/__fixtures__/*.golden.json` へ出力。TSエンジンのテストで相対誤差1e-9以内の一致を検証  
2. **プロパティベーステスト(fast-check)**: 成長率上昇→EV単調増加 / 成功確率100%のrNPV=通常NPV / 割引率上昇→現在価値減少 / 希薄化後持分総和=100% など、セクター毎に10前後の性質

受入条件: golden一致 \+ 全プロパティGreen。境界値(ゼロ成長、成功確率0/1、単一ラウンド等)を含む。

## 8\. 非機能要件

- レスポンシブ(デスクトップ主)、初期ロード\<3秒(コード分割)  
- エンジン部テストカバレッジ原則100%  
- ダーク/ライト切替時のレイアウトシフトなし(装飾トークンのみ切替、構造は共通)  
- 両テーマでWebフォント(幾何学サンセリフ/丸ゴシック)を使用するため、フォントは事前ロード+`font-display: swap` でFOUTを最小化  
- localStorage消失対策としてJSONバックアップ導線を目立たせる
