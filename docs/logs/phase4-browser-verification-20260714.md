## 3. 検証中に見つかった自動化スクリプト側の問題(製品側の不具合ではない)

検証を自動化するスクリプトを書く過程で、以下3点はいずれも**スクリプトの実装ミス**であり、アプリ自体の欠陥ではないことを個別の再テストで確認した。

1. **ドライバー毎δ変更の再計算チェックの偽陰性**: トルネード表はspan降順で自動ソートされるため、「1番上の行」を編集すると、その行のspanが変化して表内の順位が変わる。編集後に再び「1番上の行」を読むと、それは編集前とは別の(たまたま同じspan値だった)行を指してしまい、あたかも変化していないように見えた。ドライバーラベルで行を追跡するよう修正して再検証したところ、正しく再計算されていることを確認した(span 438百万円→110百万円)。
2. **v1移行テスト・エクスポート/インポートテストの連鎖失敗**: `localStorage` を `page.evaluate` で直接書き換えても、HashRouterのハッシュのみが変わる遷移(`page.goto`)はフルリロードを伴わないため、Zustandストアの読み込み済み状態(`isLoaded: true`)がそのまま残り、直接書き換えた内容を拾えなかった。`location.hash` 設定後に明示的な `page.reload()` を挟むことで解消し、正しくv1移行・エクスポート/インポートとも検証できた。
3. **テーマ切替ボタンの検出失敗**: `ThemeToggle` ボタンは `aria-label="テーマ切替"` を持つため、アクセシブルネームは可視テキスト(「ライトモードへ切替」等)ではなく `aria-label` の値になる。アクセシブルネームでの照会に修正し、正しくクリック・トグルできることを確認した。

いずれもテストスクリプトの記述ミスであり、製品コードの修正は不要と判断した。

## 4. C-4「ダークテーマのコントラスト不足」の再調査結果: 誤検知と判断

`docs/logs/phase4-test-rerun-instructions-20260714.md` §C-4 で「ダークテーマの見出し・本文の文字色が背景色に近く、コントラスト不足の疑い」と記録されていた点を実際に再現・調査した。

- ページ読み込み直後(0秒時点)のスクリーンショットでは、確かに見出し等がほぼ判読できないほど暗く見えた
- しかし3秒待機後に再度スクリーンショットを撮ると、同じ要素が完全に判読可能な高コントラスト表示になっていた
- 該当要素(`h1`)の実際の計算後スタイルは `color: rgb(215, 242, 240)`(明るいシアン系白)、背景 `rgb(5, 8, 10)`(ほぼ黒)で、コントラスト比は非常に高い(WCAG AA水準を大きく上回る)

これは静的なコントラスト不足ではなく、**ダークテーマの意図された演出(requirements-rev5.md §5 のS.A.C./PSYCHO-PASS様式「スキャン演出」に該当すると推測される、ページ読み込み時のテキスト漸次表示アニメーション)が完了する前にスクリーンショットを撮ったことによる見かけ上の現象**と判断する。**製品側の修正は不要**。手順書・メモリ内の当該記述は本書により訂正する。

**訂正(2026-07-14 査定時)**: 上記の根本原因の推測(意図された演出)は誤り。実際は `data-theme` 属性が React マウント後の useEffect で付与されるための**初回描画テーマ適用フラッシュ(FOUC)**である(スキャン演出は Phase 6 で実装予定であり現存しない)。「静的なコントラスト不足ではない・Phase 4 の修正不要」の結論は維持。詳細と対応方針は `docs/logs/phase4-verification-assessment-20260714.md` §2 を参照。
